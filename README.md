# Healthper Portfolio

> 헬스장 운동기구에 IoT 기술을 적용하여 사용자에게 운동 데이터 측정, 자동 무게 조절, 1RM 기반 점수 계산을 지원하여
   사용자에게 안전하게 운동을 할 수 있게 도움을 주고 경쟁심리를 유도하여 사용자의 운동 흥미를 높임

### 데이터 출처

Healthper에서 사용하는 데이터는 운동기구에 부착된 IoT 센서와 사용자의 운동 입력값에서 발생합니다.

### 데이터 수집 흐름

사용자가 운동을 시작하면 하나의 운동 세션이 생성되고, 센서 데이터는 해당 세션에 연결되도록 설계했습니다.

flowchart LR
    A[QR인식/사용자 운동 시작] --> B[sessionId 생성]
    B --> C[운동기구 equipmentId 연결]
    C --> D[센서 sensorId 연결]
    D --> E[ADXL345 센서 데이터 수집]
    E --> F[운동 수행 데이터 정리]
    F --> G[1RM 기반 점수 계산]
    G --> H[사용자별 운동 기록 저장]


데이터 처리 기준

센서 데이터가 여러 기구에서 동시에 발생할 수 있기 때문에, 각 데이터에 식별값을 함께 저장하는 구조로 설계했습니다.

항목	역할
userId	어떤 사용자의 운동인지 구분
equipmentId	어떤 운동기구에서 발생한 데이터인지 구분
sensorId	어떤 센서에서 발생한 데이터인지 구분
sessionId	한 번의 운동 기록 단위를 구분
timestamp	데이터가 발생한 시간 기록

이 구조를 통해 센서 데이터가 섞이더라도 특정 사용자, 특정 운동기구, 특정 운동 세션에 해당하는 데이터만 점수 계산에 사용할 수 있도록 했습니다.

데이터 정제 및 점수화

수집된 센서 데이터와 운동 수행 데이터는 점수 계산에 필요한 값으로 정리됩니다.

처리 항목	설명
반복 횟수	운동 수행 중 감지된 움직임을 기준으로 계산
운동 무게	사용자가 선택한 무게 값 사용
자세 완성도	센서 기반 움직임 안정성을 기준으로 반영
예상 1RM	Epley 공식을 사용해 무게와 반복 횟수를 하나의 기준으로 환산
최종 점수	1RM 기반 기본 점수에 자세 완성도 보정값을 적용
예상 1RM = 운동 무게 × (1 + 반복 횟수 ÷ 30)
최종 점수 = 기본 점수 × (0.9 + 자세완성도 / 100 × 0.1)
Data 관점에서의 프로젝트 의의

Healthper는 단순히 운동 결과를 저장하는 것이 아니라, 센서 데이터와 사용자 운동 데이터를 세션 단위로 구분하고 점수화하는 구조를 설계한 프로젝트입니다.

특히 userId, equipmentId, sensorId, sessionId를 기준으로 데이터를 분리하여 여러 센서 데이터가 동시에 발생할 수 있는 환경에서도 사용자별 운동 기록을 관리할 수 있도록 설계했습니다.

## Features
- 자동 무게 조절: 운동 결과에 따라 다음 운동 무게를 증가, 유지, 감소하도록 설계
- 점수 측정 알고리즘: Epley 1RM 공식을 기반으로 무게와 반복 횟수를 점수화
- 센서 기반 측정: ADXL345 가속도 센서를 활용해 운동 수행 데이터를 측정

## Architecture
Healthper는 헬스장 내 모든 운동기구를 자동화하는 시스템이 아니라,
반복 동작이 명확한 일부 기구에 IoT 센서를 적용하여
운동 데이터를 점수화하는 오락형 피트니스 프로토타입 이기 때문에
프로젝트의 주 목적은 정밀한 운동 처방보다 사용자의 경쟁심과 흥미를 유도하는 데 있습니다.
따라서 전체 헬스기구를 대상으로 하기보다는
랫풀다운, 레그프레스, 체스트프레스처럼 동작이 일정하고 점수화가 쉬운 2~3개의 기구를 우선 대상으로 설정했습니다.
사용자는 운동 결과를 점수로 확인하고, 랭킹을 통해 다른 사용자와 경쟁할 수 있습니다.
이를 통해 단순 반복 운동에 게임적 요소를 더해 헬스장 안에서 오락기기를 사용하는 듯한 흥미로운 운동 경험을 제공하는 것을 목표로 했습니다.

## 기술스택
Hardware Raspberry Pi, Step Motor, ADXL345
Backend Node.js, Express 
Communication  Socket.io 
Frontend HTML, CSS, JavaScript 
Algorithm Epley 1RM Formula, Weight Score Logic

1. Rapberry Pi 선택
항목
상황 Step Motor와 ADXL345와 연결
대안 Arduino, ESP32, Raspberry Pi
선택 센서·모터 제어 가능 여부, Node.js 서버 실행 가능 여부, 웹 화면과의 연동 편의성
결정 Raspberry Pi를 메인 제어 장치로 선택했습니다.
결과 ADXL345 센서 데이터 수집, Step Motor 제어, Node.js 서버 실행을 하나의 장치에서 처리할 수 있어 하드웨어와 웹 서버 연동 구조를 단순화할 수 있었습니다.

2. Step Motor
항목
상황 무게 핀 자동 조절을 위한 이동
대안 서보모터 / step Motor
선택 공간 확보성, 가격
결정 Step Motor 선택
결과 협소한 헬스장에서 최소한의 공간으로 기기 설치 가능

3. ADXL345 선택
항목	내용
상황	운동 횟수와 자세 측정 필요
대안	초음파 센서 / 리미트 스위치 / ADXL345
선택 기준	설치 편의성, 비용, 측정 가능 데이터
결정	ADXL345 선택
결과	하나의 센서로 기울기와 움직임 모두 측정 가능

4. Node.js 선택
항목	내용
상황	실시간 운동 데이터 처리 필요
대안	Spring Boot / Node.js
선택 기준	개발 속도, Socket.io 지원, JavaScript 사용
결정	Node.js 선택
결과	프론트·백엔드를 JavaScript로 통일 가능

5. Express 선택
항목 
상황 Node.js 서버에서 사용자 요청을 처리, 운동 데이터 저장·점수 계산·화면 연결을 위한 기본 서버 구조가 필요
대안 Node.js 기본 HTTP 모듈, Express, Fastify
선택 구현 난이도, 라우팅 처리 편의성, 정적 파일 제공 가능 여부, 팀 프로젝트에서 이해하기 쉬운 구조
결정 Express 선택
결과 복잡한 서버 설정 없이 라우팅, 정적 파일 제공, 데이터 처리 흐름을 빠르게 구성할 수 가능, 센서 데이터와 웹 화면을 연결하는 서버 구조를 단순하게 설계 가능


10. 1RM 점수 알고리즘
